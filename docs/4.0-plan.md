# UMM 4.0 版本计划

## 一、当前状态分析

### 1.1 样式系统现状

**Tailwind CSS v4 使用情况**：
- 总组件数：165个
- 使用Tailwind：112个（67.9%）
- 未使用Tailwind：53个（32.1%）

**样式系统分裂问题**：
- 4种CSS方法论共存：Tailwind、Shadow DOM、Vue scoped、程序化注入
- 自定义CSS文件：2,556行（7个文件）
- Shadow DOM内容脚本：8个组件，完全独立样式系统

**关键问题**：
1. `design-tokens.css`与`typography.css`存在变量冲突（`--space-*`重复定义）
2. `global.ts`中硬编码颜色值未与CSS变量绑定
3. Shadow DOM组件无法使用全局Tailwind类

### 1.2 组件架构现状

**组件分布**：
- `components/ui/`（shadcn组件库）：141个
- `entrypoints/options/`（设置页面）：9个
- `components/`（业务组件）：5个
- `entrypoints/popup/`（弹窗）：2个
- `entrypoints/douban-*.content/`（内容脚本）：8个

**潜在重复逻辑**：
- 多个组件可能包含相似的业务逻辑
- composables可能缺乏统一抽象
- 跨组件的数据处理逻辑可能重复

## 二、4.0 版本目标

### 2.1 核心目标

1. **样式系统统一**：将Tailwind CSS v4作为主要样式方案，减少自定义CSS
2. **组件逻辑抽象**：建立shared子模块，统一可重复的业务逻辑
3. **性能优化**：减少CSS文件大小，优化组件加载性能
4. **开发体验**：提供更一致的开发模式和工具链

### 2.2 量化指标

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| Tailwind组件占比 | 67.9% | >85% |
| 自定义CSS行数 | 2,556行 | <1,500行 |
| 组件重复逻辑 | 待评估 | <5% |
| CSS文件大小 | 待测量 | 减少30% |

## 三、具体改进计划

### 3.1 样式系统统一（优先级：高）

**阶段1：消除变量冲突**
- 合并`design-tokens.css`和`typography.css`中的重复变量
- 建立单一设计令牌源（建议保留`design-tokens.css`）
- 更新所有引用点

**阶段2：Shadow DOM组件优化**
- 评估是否可以将部分Shadow DOM组件迁移到全局Tailwind
- 对于必须使用Shadow DOM的组件，建立统一的CSS变量表
- 将`global.ts`中的硬编码颜色提取为CSS变量

**阶段3：Tailwind配置优化**
- 利用Tailwind v4的`@theme`指令扩展配置
- 将自定义设计令牌映射到Tailwind配置
- 减少自定义CSS类的使用

### 3.2 组件逻辑抽象（优先级：高）

**阶段1：重复逻辑识别**
- 扫描所有组件，识别相似的业务逻辑
- 分析composables的使用情况
- 评估数据处理、API调用、状态管理的重复度

**阶段2：shared模块设计**
- 设计shared子模块架构
- 定义统一的接口和类型
- 建立组件间的数据共享机制

**阶段3：逐步重构**
- 按优先级重构重复逻辑
- 建立组件测试套件
- 确保重构不破坏现有功能

### 3.3 性能优化（优先级：中）

**CSS优化**：
- 启用Tailwind的PurgeCSS功能
- 移除未使用的CSS类
- 压缩CSS文件

**组件优化**：
- 实现组件懒加载
- 减少组件重渲染
- 优化事件处理

### 3.4 开发体验（优先级：中）

**工具链优化**：
- 提供组件开发文档
- 建立组件测试规范
- 优化构建配置

**代码规范**：
- 制定组件开发规范
- 建立代码审查标准
- 提供开发示例

## 五、风险评估

### 5.1 技术风险

- **Shadow DOM迁移风险**：部分组件可能无法完全迁移到Tailwind
- **兼容性风险**：重构可能影响现有功能
- **性能风险**：优化可能引入新的性能问题

### 5.2 缓解措施

- 充分测试每个阶段的改动
- 保持向后兼容性
- 监控性能指标
- 建立回滚机制
